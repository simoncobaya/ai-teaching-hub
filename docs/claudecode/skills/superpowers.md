# 🦸 Superpowers — 给 AI 装上"软件工程超能力"

> Superpowers 是由 [Jesse Vincent](https://github.com/obra) 打造的开源插件，GitHub 上获得 **204K+ Star**，官方市场安装量 **82 万+**。

---

## 🤔 它解决了什么问题？

你有没有遇到过这种情况：

> 让 AI 写一个功能，它二话不说就开始写代码，写了 200 行发现方向完全错了……

**Superpowers 就是来解决这个问题的。** 它给 AI 套上了一套"软件工程纪律"——

```
❌ 没有 Superpowers：你说"做吧" → AI 直接写代码 → 方向错了 → 重来
✅ 有 Superpowers：  你说"做吧" → AI 先提问搞清楚 → 写计划 → 你确认 → 再动手
```

就像盖房子前要先画图纸 🏗️，写作文前要先列提纲 📝——Superpowers 让 AI 养成"**先想清楚再动手**"的习惯。

## 📦 安装

在 Claude Code 中输入：

```
/plugin install superpowers@claude-plugins-official
```

安装后**不需要任何配置**。Superpowers 会自动启动——每次打开 Claude Code，它就默默在后台运行了。

## 🔄 核心工作流

Superpowers 把"开发一个功能"拆成了 **7 个标准步骤**。AI 会一个接一个地走，不会跳步：

```
你："帮我做一个猜数字游戏"

  Step 1  🧠 头脑风暴      AI 先提问："游戏规则是什么？几个玩家？什么平台？"
         brainstorming     搞清楚需求，不急着写代码

  Step 2  📝 写计划         AI 把任务拆成小步骤："1.创建HTML文件 2.写游戏逻辑..."
         writing-plans      每个步骤 2-5 分钟能完成

  Step 3  📦 隔离工作区      AI 创建独立的分支，不影响你已有的代码
         using-git-worktrees

  Step 4  🤖 子Agent执行     AI 派出"小弟"一个个完成任务
         subagent-driven-dev 每个小弟只做一个步骤，做完就审查

  Step 5  🧪 TDD 测试先行   写代码前先写测试——"红→绿→重构"
         test-driven-dev     测试不通过？先别写实现代码！

  Step 6  🔍 代码审查        AI 自己审查自己的代码
         requesting-review   找出问题 → 修复 → 再审查

  Step 7  🏁 收尾            测试全绿？合并代码还是提交 PR？你决定
         finishing-branch    清理工作区，完美收工
```

> 💡 **用盖房子来比喻**：头脑风暴 = 和业主沟通需求 → 写计划 = 画施工图 → 工作区隔离 = 围起工地 → 子Agent执行 = 不同工种的师傅各干各的 → TDD = 每砌一面墙就检查 → 代码审查 = 监理验收 → 收尾 = 打扫工地交房

## 📋 14 个技能详解

Superpowers 包含 14 个 Skill，分为四大类：

---

### 🧠 思考类（动手前先想清楚）

#### 1. 🎨 brainstorming（头脑风暴）

| 项目 | 内容 |
|------|------|
| Slash | `/brainstorming` |
| 一句话 | **在写任何代码之前，先搞清楚你到底要做什么** |
| 触发时机 | 你说"帮我做一个XX"、"添加XX功能"时自动触发 |

**AI 会怎么做：**
1. 探索项目背景——看看你项目里已经有什么
2. 问你问题——**一次只问一个**，不轰炸你
3. 提出 2-3 种方案，推荐最优的
4. 写出设计文档，让你确认
5. 你点头了，才进入下一步

> ⚠️ **硬性规定**：设计没被确认之前，**绝不写代码**。

**对学生来说：** 这个技能教你"想清楚再动手"的工程思维。AI 会像老师一样引导你思考：你的游戏规则是什么？用户怎么操作？界面长什么样？

---

#### 2. 📐 writing-plans（写计划）

| 项目 | 内容 |
|------|------|
| Slash | `/writing-plans` |
| 一句话 | **把大任务拆成 2-5 分钟能完成的小步骤** |
| 触发时机 | 头脑风暴完成后自动触发 |

**AI 会怎么做：**
1. 读取设计文档
2. 把功能拆成原子任务——每个任务 2-5 分钟
3. 写出每个任务需要的**具体代码**、**测试命令**、**预期输出**
4. 标注任务之间的依赖关系
5. 让你确认整个计划

> 🚫 **绝对禁止**：计划里不能有"TODO"、"待补充"、"适当添加错误处理"这种模糊描述。

**对学生来说：** 这就是"写作文先列提纲"的编程版。大问题拆成小问题，逐个击破。

---

### 🛠️ 执行类（按计划动手）

#### 3. 🤖 subagent-driven-development（子 Agent 执行）

| 项目 | 内容 |
|------|------|
| Slash | `/subagent-driven-development` |
| 一句话 | **每个任务派一个"全新大脑"去执行，互不干扰** |
| 触发时机 | 计划确认后自动触发（推荐方式） |

**AI 会怎么做：**
1. 创建 TODO 列表（对应计划中的每个任务）
2. 对每个任务：
   - 派一个**全新的子 Agent**（干净上下文，不受前面对话干扰）
   - 子 Agent 执行 TDD（写测试→写代码→通过→提交）
   - 子 Agent **自我审查**
   - AI 再派一个**审查 Agent** 检查子 Agent 的工作
   - 有问题？派修复 Agent 解决
3. 所有任务完成后，做**全分支最终审查**

> 💡 这就像老师把班级分成小组，每组负责不同的任务，每组完成后互相检查。

---

#### 4. 📋 executing-plans（内联执行）

| 项目 | 内容 |
|------|------|
| Slash | `/executing-plans` |
| 一句话 | **在不支持子 Agent 的环境里，自己一步步执行计划** |
| 触发时机 | 计划确认后，环境不支持子 Agent 时使用 |

和 `subagent-driven-development` 的区别：不派小弟，AI 自己一步步做。Superpowers 推荐用子 Agent 方式（效果更好），这只是备选方案。

---

#### 5. 🌿 using-git-worktrees（隔离工作区）

| 项目 | 内容 |
|------|------|
| Slash | `/using-git-worktrees` |
| 一句话 | **在独立分支上工作，不影响主代码** |
| 触发时机 | 开始执行计划前自动触发 |

**AI 会怎么做：**
1. 创建独立的工作区（git worktree）
2. 自动运行项目初始化（`npm install` 等）
3. 验证测试基线——确保当前代码是好的

> 💡 就像做化学实验前戴上手套——保护你的主代码不被"污染"。

---

#### 6. 🤝 dispatching-parallel-agents（并行调度）

| 项目 | 内容 |
|------|------|
| Slash | `/dispatching-parallel-agents` |
| 一句话 | **把不相关的任务同时派给多个 Agent，节省时间** |
| 触发时机 | 有 2+ 个互不依赖的任务时自动触发 |

> 💡 就像数学作业和语文作业可以同时做，不用等一个做完再开始另一个。

---

#### 7. 🏁 finishing-a-development-branch（收尾）

| 项目 | 内容 |
|------|------|
| Slash | `/finishing-a-development-branch` |
| 一句话 | **功能做完了，决定怎么处理：合并？提 PR？还是放弃？** |
| 触发时机 | 所有任务完成、测试全绿时自动触发 |

**AI 会怎么做：**
1. 验证所有测试通过
2. 检测当前环境（普通分支？工作区？）
3. 让你选择：合并到主分支 / 推送并创建 PR / 暂存 / 丢弃
4. 执行你的选择，清理工作区

> 💡 就像作业写完了，老师问你：交上来？先给同桌检查？还是重写？

---

### ✅ 质量类（保证代码质量）

#### 8. 🧪 test-driven-development（测试驱动开发）

| 项目 | 内容 |
|------|------|
| Slash | `/test-driven-development` |
| 一句话 | **"没有失败的测试，就不写实现代码！"** |
| 触发时机 | 写任何功能代码前自动触发 |

**铁律三步走 🔴🟢🔄：**
1. 🔴 **RED（红）**：先写一个失败的测试，亲眼看到它失败
2. 🟢 **GREEN（绿）**：写最少的代码让测试通过，亲眼看到它通过
3. 🔄 **REFACTOR（重构）**：清理代码，保持整洁，测试仍然是绿的

> 🚫 如果先写了实现代码再补测试？**删掉重来。**

**对学生来说：** TDD 就像"做数学题先写验算步骤"——你知道答案应该是什么，然后去验证你的计算是否正确。这是一种非常好的学习习惯！

---

#### 9. 🔍 requesting-code-review（请求代码审查）

| 项目 | 内容 |
|------|------|
| Slash | `/requesting-code-review` |
| 一句话 | **让 AI 派一个"审查官"来检查代码质量** |
| 触发时机 | 完成一个任务后、合并前自动触发 |

审查官会检查：
- 代码是否符合计划？
- 有没有明显的 Bug？
- 架构是否合理？
- 测试是否完整？

问题分三级：🔴 致命（必须修）、🟡 重要（应该修）、🟢 小建议（看情况）。

---

#### 10. 📨 receiving-code-review（处理审查反馈）

| 项目 | 内容 |
|------|------|
| Slash | `/receiving-code-review` |
| 一句话 | **收到审查意见后，先验证再改，不要盲目同意** |
| 触发时机 | 收到代码审查反馈时自动触发 |

**核心原则**：先验证再实现。不是所有的审查意见都是对的——用技术判断，不要为了"好相处"而盲目同意。

---

#### 11. ✅ verification-before-completion（完成前验证）

| 项目 | 内容 |
|------|------|
| Slash | `/verification-before-completion` |
| 一句话 | **"说你做完了之前，先跑一遍测试让我看看！"** |
| 触发时机 | 声称"做好了"之前自动触发 |

**铁律**：没有**新的**验证证据，就不能说"做好了"。上次跑的不算，要刚才跑的才算。

**对学生来说：** 这就是"交作业前先检查一遍"——不要写完就交，先自己验证一下对不对。

---

### 🔧 调试与元技能

#### 12. 🐛 systematic-debugging（系统化调试）

| 项目 | 内容 |
|------|------|
| Slash | `/systematic-debugging` |
| 一句话 | **遇到 Bug 不要乱猜，按 4 步流程科学排查** |
| 触发时机 | 遇到任何 Bug、测试失败、异常行为时自动触发 |

**四阶段排查法：**
1. 🔎 **找根因**：仔细读报错信息，重现问题，检查最近改动
2. 📊 **模式分析**：找到能正常工作的类似代码，对比差异
3. 💡 **假设验证**：提出一个假设，用最小改动验证
4. 🔧 **实施修复**：先写失败的测试（证明 Bug 存在），再修复

> 🚫 修了 3 次还没修好？停下来，可能整体设计有问题。

**对学生来说：** 这就是科学方法的编程版——观察 → 假设 → 实验 → 结论。不靠猜，靠证据。

---

#### 13. ✍️ writing-skills（编写技能）

| 项目 | 内容 |
|------|------|
| Slash | `/writing-skills` |
| 一句话 | **用 TDD 的方式创建和修改 Skill** |
| 触发时机 | 创建或修改 Skill 时触发 |

铁律：**没有失败的测试，就不写 Skill。** 写好 Skill 后，用压力测试验证它真的有效。

---

#### 14. 🧭 using-superpowers（使用 superpowers）

| 项目 | 内容 |
|------|------|
| Slash | `/using-superpowers` |
| 一句话 | **告诉 AI "你现在有 superpowers 了，遇事先查技能"** |
| 触发时机 | **每次会话启动时自动加载**（不需要你手动调） |

这是 superpowers 的"大脑"——它建立了一个规则：**任何任务开始前，AI 必须先检查有没有对应的 Skill 可以用**。哪怕只有 1% 的可能性匹配，也必须先看 Skill。

它还包含一个"借口识别表"——AI 如果试图跳过 Skill，会被这张表纠正：
- "这看起来很简单" → 不对，简单任务也要查 Skill
- "我已经知道怎么做了" → 不对，Skill 可能有更好的方法
- "用户很着急" → 不对，跳过流程反而更慢

## 🪝 SessionStart Hook — 自动启动的秘密

Superpowers 安装后不需要任何配置就能工作，因为它有一个 **SessionStart Hook**：

```
会话启动 / clear / compact
    ↓
hooks/session-start 脚本运行
    ↓
读取 using-superpowers/SKILL.md 的完整内容
    ↓
注入到 Claude 的上下文中
    ↓
Claude 获得"superpowers 意识"
```

这个 Hook 在三个时机触发：
- 🔵 每次新会话启动
- 🔵 每次 `/clear`（清空对话）
- 🔵 每次 `/compact`（压缩对话）

所以无论你什么时候打开 Claude Code，superpowers 都在默默运行。

## 🎮 适合学生吗？

| 技能 | 适合初学者？ | 说明 |
|------|:--:|------|
| brainstorming | ⭐⭐⭐ | 教"想清楚再动手"，非常好的习惯 |
| writing-plans | ⭐⭐⭐ | 教"大问题拆小问题"的思维方式 |
| test-driven-development | ⭐⭐⭐ | 教"先想答案再验证"，学习方法神器 |
| verification-before-completion | ⭐⭐⭐ | 教"交作业前先检查" |
| systematic-debugging | ⭐⭐ | 教科学排查方法，对初学者略难 |
| subagent-driven-development | ⭐⭐ | 自动化执行，但对 9-12 岁可能太复杂 |
| requesting/receiving-code-review | ⭐⭐ | 审查代码是好习惯，但需要一定基础 |
| using-git-worktrees | ⭐ | Git 高级操作，对初学者较难 |
| finishing-a-development-branch | ⭐ | 同上 |
| dispatching-parallel-agents | ⭐ | 高级并行调度 |
| writing-skills | ⭐ | 需要先理解 Skill 系统 |

> 💡 **建议**：对于 4-6 年级学生，superpowers 最大的价值在前 4 个技能（头脑风暴、写计划、TDD、完成前验证）。它们教的是**思维方式**，而不是技术细节。后面偏工程化的技能，可以在熟练掌握基础后再探索。
