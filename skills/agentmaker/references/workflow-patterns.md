# 工作流设计模式参考

> 本文档供 `/agentmaker` Skill 阶段2使用，基于7个竞赛案例的工作流分析提炼  
> 包含4种基础设计模式 + Subflow三原则详解 + 案例实战剖析

---

## 一、基础工作流模式

### 1.1 串联式（Pipeline）

**模式描述**：节点线性串联，前一节点的输出作为后一节点的输入，形成处理流水线。

**适用场景**：
- 处理流程步骤明确、顺序固定
- 无复杂分支逻辑
- 数据流是单向的（A→B→C→D）

**典型节点构成**：
```
[开始] → [LLM处理1] → [LLM处理2] → ... → [LLM处理N] → [输出/结束]
```

**典型案例**：案例6"二手交易避坑助手"的初筛和复核子流程

```yaml
# 初筛流程（5个节点纯串联）
开始 → price_analysis(LLM) → tactics_analysis(LLM) 
     → method_analysis(LLM) → risk_assessment(LLM) → 结束
```

**搭建顺序建议**：从左到右依次搭建，逐个节点测试通过后再添加下一个。

**何时使用**：
- ✅ 节点预估 ≤ 6 个且逻辑单一
- ✅ 不需要根据中间结果改变后续流程
- ❌ 需要根据AI输出做分支选择时，改用分支式

---

### 1.2 分支式（Router）

**模式描述**：通过条件选择器（condition节点）根据LLM输出结果将流程路由到不同处理路径。

**适用场景**：
- 需要根据AI判断结果走不同处理逻辑
- 用户意图识别后分流
- 风险评估后分级处理

**典型节点构成**：
```
[开始] → [意图识别LLM] → [选择器] ┬→ [分支A: LLM处理]
                                  ├→ [分支B: LLM处理]
                                  └→ [分支C: 默认处理]
                                   ↓
                              [变量聚合] → [输出/结束]
```

**典型案例**：案例1的意图分流

```yaml
# 一入口四路径
开始 → 意图分类(LLM) → 选择器(condition)
  ├── study → 出题流程(subflow)
  ├── explain → 知识讲解(LLM)  
  ├── progress → 进度查询(LLM)
  └── 兜底 → 闲聊应答(LLM)
→ 变量聚合 → 结束
```

**判断节点位置**：紧接在"分析/分类"类LLM节点之后。

**搭建顺序建议**：先搭建主路径（最常见分支），验证通过后再添加其他分支。

**何时使用**：
- ✅ 需要根据AI输出做不同处理
- ✅ 用户有多种交互意图
- ✅ 这是**必须的**——竞赛要求至少1个基于大模型输出的判断节点

---

### 1.3 循环式（Loop）

**模式描述**：通过Loop节点或条件选择器形成回路，让某段流程重复执行直到满足条件。

**适用场景**：
- AI生成内容需要反复校验直到合格
- 多轮问答/对话
- 需要重试机制的场景

**两类实现方式**：

#### A. Loop节点方式（推荐，案例1）

```
[开始] → [Loop条件] → [处理LLM] → [校验代码] → [条件选择]
                         ↑                                  ↓
                         └── (不通过，继续循环) ←── [不通过]  
                                                       [通过/超限] → [终止循环] → [结束]
```

**案例1出题校验循环**（最多10次重试）：
- LLM生成题目 → 代码节点校验（格式/内容/安全性）
- 通过 → 终止循环，输出题目
- 不通过且重试≤2次 → 继续循环
- 不通过且重试>2次 → 终止循环，使用兜底题目

#### B. 条件回路方式（案例7面试循环）

```
[开始] → [提问LLM] → [收集答案] → [分析LLM] → [条件选择]
              ↑                                       ↓
              └── (还有问题) ←←←←←←←←←← [剩余>0]
                                              [剩余=0] → [下一阶段] → [结束]
```

**使用时注意**：
- Loop节点需要设置**最大循环次数**，防止死循环
- 需要设置**兜底逻辑**（超过重试次数后的处理方案）
- 代码校验节点配合使用效果最佳

---

### 1.4 多智能体协同式（Multi-Agent）

**模式描述**：多个LLM节点分别承担不同角色和职责，分工协作完成复杂任务。

**适用场景**：
- 任务需要多角度分析
- 需要模拟不同角色的专业判断
- 复杂决策需要分步推理

**典型节点构成**：
```
[开始] → → [Agent A: 预处理] → [Agent B: 维度分析1] → [Agent C: 维度分析2]
         → [Agent D: 综合判断] → [Agent E: 结果解释] → [结束]
```

**案例5（factcheck）7个Agent角色示例**：

| 角色 | 节点名 | 职责 |
|------|--------|------|
| 预处理员 | preprocess | 整理输入信息，提取关键字段 |
| 来源侦探 | source_analysis | 分析信息来源可信度 |
| 证据审查员 | evidence_analysis | 检查文本/图片证据质量 |
| 情绪检测员 | credibility_judge | 综合评估可信度等级 |
| 高风险详解员 | high_risk_detail | 解释高风险判断理由 |
| 中风险建议员 | medium_risk_advice | 给出中等风险处理建议 |
| 低风险确认员 | low_risk_confirm | 确认安全并解释原因 |

**多智能体设计的提示词要点**：
- 每个Agent有独立的角色定义和人设
- 输入输出格式需要在提示词中明确定义
- Agent之间通过变量传递信息

**注意**："多智能体"在Coze中体现为**多个LLM节点**，不一定需要多个Bot。同一Bot内的多个LLM节点承担不同角色，也属于多智能体协同。

## 二、Subflow 三原则详解

Subflow（子流程）是Coze中将一部分可复用的工作流逻辑封装为独立workflow，被上层chatflow/workflow调用的机制。

### 原则一：单向依赖（下层不知道上层，上层调用下层）

**核心思想**：Subflow之间形成严格的单向依赖树，只能上层调用下层，不允许下层回调上层，也不能形成循环依赖。

**依赖树示例**（案例1）：
```
adaptive_learning_chat (chatflow入口)
    └── adaptive_learning (第1层)
        ├── adaptive_question_generation (第2层-a)
        │   ├── knowledge_base_retrieval (第3层-a)
        │   └── question_generation (第3层-b)
        └── answering_process (第2层-b)
```

**违规示例**（应避免）：
```
❌ A调用B，B又调用A（循环依赖）
❌ A调用B，B调用C，C调用A（间接循环）
❌ 叶子节点subflow试图调用上层subflow
```

**实现要点**：
- 画出依赖树，确认没有环
- 上层主动传数据给下层（通过"输入变量"）
- 下层处理完返回结果给上层（通过"输出变量"）
- 下层不应该"知道"是谁调用了自己

---

### 原则二：先拆后合（先拆叶子，再组合上层）

**核心思想**：从最底层、最独立的子任务开始拆分，然后逐层向上组合，形成完整功能。

**拆解步骤**：

1. **列出所有子任务**：把所有要做的事情列出来
2. **识别最基础的任务**（叶子节点）：哪些任务不需要依赖其他任务的结果？
3. **先实现叶子subflow**：把每个叶子subflow独立实现并测试
4. **逐层组合**：把叶子subflow组合成中层subflow，再把中层组合成顶层

**案例1拆解过程**：

| 步骤 | 拆解内容 | 结果 |
|:---:|----------|------|
| 1 | 列出所有子任务 | 意图识别、知识检索、出题、答题评判、讲解、进度查询等 |
| 2 | 识别叶子节点 | 知识库检索（只需输入学科，不依赖其他任务） |
| 3 | 实现叶子 | `knowledge_base_retrieval`（3个知识库+学科路由） |
| 4 | 下一层叶子 | `question_generation`（调用LLM出题+校验） |
| 5 | 组合中层 | `adaptive_question_generation` = 知识检索 + 出题 + 能力评估 |
| 6 | 组合另一中层 | `answering_process` = 题目格式化 + 收集答案 + 评判 |
| 7 | 组合顶层 | `adaptive_learning` = 意图路由 + 出题子流程 + 答题子流程 |

**验证方法**：最底层的subflow应该可以独立运行和测试，不依赖其他subflow。

---

### 原则三：接口先行（先定义输入输出，再设计内部实现）

**核心思想**：为每个subflow先定义清晰的"合同"（输入变量和输出变量），就像搭积木时先确定接口形状再填充内部。

**接口定义模板**：

```yaml
# Subflow: xxx_subflow
# 职责：一句话描述这个subflow做什么

输入变量：
  - input_text (String): 用户输入的文本
  - user_level (String): 用户能力等级，可选值：beginner/intermediate/advanced
  - context_data (Object, 可选): 上下文数据

输出变量：
  - result (String): 处理结果的文本输出
  - status (String): 处理状态，可选值：success/failed/retry
  - score (Number): 评分结果（0-100）

内部逻辑（简要）：[此处描述核心处理步骤]
```

**案例1接口定义示例**：

```yaml
# Subflow: question_generation
# 职责：根据学科知识点和能力等级，生成自适应难度的题目

输入：
  - subject (String): 学科，math/chinese/english
  - knowledge_point (String): 具体知识点，如"一元一次方程"
  - difficulty (String): 难度等级，beginner/intermediate/advanced
  - question_count (Number): 出题数量

输出：
  - questions (String): 生成的题目文本
  - status (String): success=出题成功, failed=出题失败
  - retry_count (Number): 重试次数

内部逻辑：
  1. LLM根据subject+knowledge_point+difficulty生成题目
  2. 代码节点校验格式和内容完整性
  3. 不通过则重新生成（最多10次）
  4. 超限使用兜底题目
```

**为什么要接口先行**：
- 拆分复杂任务时，先确定"信息怎么流动"比"具体怎么处理"更重要
- 接口定义好了，上层和下层的subflow可以并行开发
- 方便后续修改内部实现而不影响调用方

## 三、复杂工作流判定与拆解引导

### 3.1 何时需要拆解为Subflow？

满足以下**任一条件**，建议使用Subflow拆解：

| 条件 | 说明 |
|------|------|
| 节点预估 > 6个 | 节点太多导致主工作流难以理解 |
| 涉及多条业务路径 | 分支多，每种路径的处理逻辑差异大 |
| 需要循环/重试 | 循环逻辑比较独立，适合封装 |
| 某段逻辑可能被复用 | 多个地方需要相同的处理 |
| 某段逻辑非常独立 | 可以解耦出来单独测试 |

### 3.2 常见拆解难点的处理方法

#### 难点1：不知道从哪里开始拆
- **方法**：先不考虑拆，列出所有需要做的事情（ALL tasks），然后按"是否需要依赖其他结果"排序
- **输出**：任务清单 + 依赖关系图

#### 难点2：循环依赖
- **症状**：A需要B的结果，B需要A的结果
- **方法**：检查是否可以把循环逻辑封装在同一个subflow内部
- **输出**：重构后的依赖树（无环）

#### 难点3：拆得太碎
- **症状**：一个简单功能被拆成了5-6个subflow
- **方法**：合并功能相近的subflow，2-3个相关叶子节点可以合并为一个中层subflow
- **规则**：每个subflow至少包含2-3个有意义的处理步骤

#### 难点4：变量传递混乱
- **症状**：不知道哪些变量需要在上层和下层的接口中定义
- **方法**：回到"接口先行"原则，为每个subflow明确写输入变量表和输出变量表

## 四、案例1实战剖析：从需求到工作流

### 4.1 案例概览

- **名称**：智学冒险 — AI自适应学习辅导智能体
- **组别**：初中组（≥10节点）
- **规模**：6个独立工作流，37个处理节点，6个判断节点，3层嵌套
- **评分维度覆盖**：全部5大维度13项指标

### 4.2 Subflow拆解过程回顾

#### 第0步：明确核心功能
"一个AI学习助手，能出题、能判题、能讲解、能查进度"

#### 第1步：列出所有子任务（发散步）
1. 理解用户意图（想学习？想查进度？还是闲聊？）
2. 识别学科和知识点
3. 从知识库检索相关知识
4. 评估学生能力水平
5. 根据能力生成自适应题目
6. 校验题目质量
7. 展示题目并收集答案
8. 评判答案正确性
9. 讲解错题
10. 记录学习进度

#### 第2步：按接口先行原则定义接口

```yaml
# 叶子层 Subflow 1: knowledge_base_retrieval
输入：subject (String), query (String)
输出：knowledge_content (String), found (Boolean)

# 叶子层 Subflow 2: question_generation  
输入：subject, knowledge_point, difficulty, count
输出：questions (String), status (String), retry_count (Number)

# 中层 Subflow A: adaptive_question_generation
输入：user_input, user_history
输出：questions (String), difficulty_level (String)
内部：knowledge_base_retrieval + 能力评估 + question_generation

# 中层 Subflow B: answering_process
输入：questions (String)
输出：answer_result (String), is_correct (Boolean)
内部：格式化题目 → 收集答案 → LLM评判 → 反馈

# 顶层 Subflow: adaptive_learning
输入：user_input, session_history
输出：response (String)
内部：意图识别 → 分支路由 → 调用中层Subflow A或B → 聚合输出
```

#### 第3步：建立依赖树

```
adaptive_learning_chat (用户入口，chatflow)
 └─ adaptive_learning
     ├─ adaptive_question_generation
     │   ├─ knowledge_base_retrieval  ← 叶子（不依赖其他）
     │   └─ question_generation       ← 叶子（依赖knowledge_base_retrieval的输出）
     └─ answering_process             ← 叶子（接收上层传来的题目）
```

#### 第4步：按"从底层往顶层"的搭建顺序

| 顺序 | 搭建内容 | 可以独立测试 |
|:---:|----------|:---:|
| 1 | knowledge_base_retrieval | ✅ 输入学科和查询，测试能否返回知识 |
| 2 | question_generation | ✅ 联调叶子1，测试出题+校验 |
| 3 | adaptive_question_generation | ✅ 组合叶子1+2，测试出题完整流程 |
| 4 | answering_process | ✅ 输入题目，测试答题评判 |
| 5 | adaptive_learning | ✅ 组合中层A+B，测试意图路由和完整流程 |
| 6 | adaptive_learning_chat | ✅ 作为Bot测试端到端交互 |

### 4.3 关键设计决策说明

| 决策 | 原因 | 效果 |
|------|------|------|
| 为什么3层嵌套？ | 知识库检索和出题是两个独立可复用能力 | 可以分别在出题和讲解场景中复用知识库检索 |
| 为什么出题用Loop？ | LLM出题可能格式不正确、内容不安全 | 通过代码校验+重试机制保证题目质量 |
| 为什么按学科分知识库？ | 知识库检索需要精确定位 | 按学科路由提高检索精确度 |
| 为什么要有兜底路径？ | 用户输入可能无法匹配任何已知意图 | 避免"我不知道"的糟糕体验 |

## 五、模式选择决策树

```
学生描述的需求
    │
    ├── 只有一种处理路径，没有分支？
    │   └── ✅ 串联式（Pipeline）
    │
    ├── 需要根据AI判断结果走不同处理？
    │   └── ✅ 串联式 + 分支式（必含条件选择器）
    │
    ├── 某段处理需要重复执行直到达成条件？
    │   └── ✅ 加入循环式（Loop或条件回路）
    │
    ├── 有多个不同的"专家"角色需要分别判断？
    │   └── ✅ 多智能体协同式（多个LLM节点不同角色）
    │
    ├── 节点很多（>6）或逻辑很复杂？
    │   └── ✅ 按Subflow三原则拆解
    │       ├── 先定义接口
    │       ├── 拆出叶子subflow
    │       └── 逐层组合
    │
    └── 记住：必须有至少1个基于LLM输出的判断节点！
```

## 六、常用节点类型速查

| 节点类型 | 用途 | 何时使用 |
|----------|------|----------|
| **LLM** | 大模型处理（文本生成、分析、判断） | 任何需要AI智能处理的地方 |
| **Condition（选择器）** | 条件分支路由 | 根据变量值选择不同路径 |
| **Code（代码节点）** | 格式校验、数据转换、规则判断 | 确定性的逻辑处理 |
| **Knowledge（知识库）** | RAG检索增强 | 需要专业知识库支持 |
| **Subflow（子流程）** | 调用另一个workflow | 复用逻辑或拆分复杂流程 |
| **Loop（循环）** | 重复执行某段流程 | 需要重试机制 |
| **Variable Merge（变量聚合）** | 合并多个分支的输出 | 分支汇流到同一出口 |
| **Question（问答节点）** | 向用户提问并等待回复 | 需要用户输入 |
