# 评分优化清单

> 本文档供 `/agentmaker` Skill 阶段1"可行性预评估"和阶段5"评分维度优化"使用  
> 5大维度13项指标逐一列出检查要点、常见失分点、优化动作

---

## 一、创新性（25%）

### 指标1：主题有新意（15%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 项目主题具有新意，避免大众化，杜绝雷同，实用价值突出 |
| **检查要点** | 1. 选题是否在常见竞赛主题之外？（非"天气预报""翻译""聊天"等）<br>2. 是否解决了某个具体群体的真实问题？<br>3. 是否可以清晰地说出"这个智能体和市面上的XXX有什么区别"？ |
| **常见失分点** | ❌ 选题过于大众化（查天气、背单词、聊天机器人）<br>❌ 功能描述模糊，说不出解决什么具体问题<br>❌ 与其他参赛作品可能雷同 |
| **优化动作** | ✅ 阶段1充分发散，至少浏览5个候选主题再决定<br>✅ 给智能体找一个具体的服务对象（"为谁解决什么问题"）<br>✅ 强调独特视角——同样的主题，不一样的切入点 |
| **阶段1预评估分数参考** | 5分=非常有新意/罕见主题；3分=有一定新意；1分=常规主题/易雷同 |

### 指标2：多学科融合（10%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 开发及设计中涉及多个学科的融合 |
| **检查要点** | 1. 涉及哪几个学科？（至少2个）<br>2. 是真正的跨学科融合还是简单拼接？<br>3. 各学科在智能体中有不可替代的作用吗？ |
| **常见失分点** | ❌ 只涉及单一学科<br>❌ 名义上多个学科，实际上只用了其中一科的常识<br>❌ 学科之间没有真正结合，各做各的 |
| **优化动作** | ✅ 选2-3个学科，设计它们需要"配合"才能完成的功能点<br>✅ 在方案设计文档中明确标注每个模块涉及哪些学科<br>✅ 示例：AI营养师 = 生物（营养学）+ 数学（卡路里计算）+ 信息（工作流） |

## 二、完整性（20%）

### 指标3：开发流程完整性（10%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 报告中包含需求分析、系统设计、开发与测试完整流程 |
| **检查要点** | 1. 需求分析：是否说明了为什么要做这个智能体？<br>2. 系统设计：是否有清晰的工作流架构图？<br>3. 开发与测试：是否有测试用例和测试结果？<br>4. PPT是否按四板块结构（需求/设计/测试/反思）组织？ |
| **常见失分点** | ❌ PPT缺少系统设计框图或工作流截图<br>❌ 需求分析过于简单（一句话带过）<br>❌ 没有测试部分或测试仅测了1-2个场景 |
| **优化动作** | ✅ PPT严格按四大板块组织：需求分析→系统设计→开发测试→反思<br>✅ 至少设计5个以上测试场景，覆盖正常和异常情况<br>✅ 工作流截图必须清晰，标注关键节点 |

### 指标4：提交内容和资源完整性（10%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 提交内容包括报告文件及报告视频等完整资源 |
| **检查要点** | 1. 是否所有交付件都已准备？（源码链接、附件1、PPT、视频、过程记录）<br>2. 文件命名是否符合规范？<br>3. 视频规格是否达标？（MP4、≥720p、<200MB） |
| **常见失分点** | ❌ 缺少过程文档/照片<br>❌ 命名不规范<br>❌ 视频超时或无法播放 |
| **优化动作** | ✅ 使用阶段6的交付件清单逐一核对<br>✅ 提前录制视频并检查格式和大小<br>✅ 过程记录用截图+文字说明（开发中的迭代、遇到的问题和解决过程） |

## 三、先进性（30%）— 最高权重维度

### 指标5：工作流开发（10%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 项目应用工作流开发实现 |
| **检查要点** | 1. 是否基于工作流平台开发？（Coze/Dify等）<br>2. 节点数是否达标？（小学≥5，初中/高中≥10）<br>3. 工作流是否体现了智能决策？（不只是简单线性处理） |
| **常见失分点** | ❌ 节点数不够（最常见的问题！）<br>❌ 工作流全是LLM节点，缺少代码、知识库、判断等多样化节点<br>❌ 流于形式：有workflow但核心决策没用到AI |
| **优化动作** | ✅ 确保节点数达标（初中/高中≥10是硬要求）<br>✅ 节点类型多样化：LLM + Code + Condition + Knowledge + Subflow<br>✅ 适当拆分节点（一个复杂处理拆成2-3步） |
| **阶段1预估** | 节点不够？回阶段1增加功能模块 |

### 指标6：多模态调用（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 项目设计开发流程中包含一个以上大模型调用 |
| **检查要点** | 1. 是否使用了多个LLM节点？（至少2个）<br>2. 不同LLM节点是否承担不同角色/功能？<br>3. 是否涉及图像识别等多模态能力？ |
| **常见失分点** | ❌ 只有一个LLM节点<br>❌ 有多个LLM但功能雷同<br>❌ 有图片输入场景但没用多模态模型 |
| **优化动作** | ✅ 设计至少2-3个LLM节点，每个有不同角色和提示词<br>✅ 如场景适合，加入图片识别或多模态分析<br>✅ 在方案文档中标注"本智能体使用了N个LLM节点，分别负责XXX" |

### 指标7：多智能体协同（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 系统中包含1个以上智能体，多智能体协同 |
| **检查要点** | 1. 是否有多个LLM节点承担不同角色？<br>2. 不同Agent之间是否有协作关系？（A的输出影响B的行为）<br>3. 是否用了Subflow做模块化拆分？ |
| **常见失分点** | ❌ 只有一个Agent角色<br>❌ 多个LLM但无协同——各自独立输出 |
| **优化动作** | ✅ 设计2-3个Agent角色，明确分工<br>✅ 让Agent的输出作为其他Agent的输入<br>✅ 使用Subflow封装独立Agent模块<br>✅ 示例：预处理Agent → 分析Agent → 决策Agent → 解释Agent |

### 指标8：智能人机交互（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 系统包含一种以上智能人机交互 |
| **检查要点** | 1. 是否支持多轮对话？<br>2. 是否有语音/图片/文字等多种交互方式？<br>3. AI是否能根据上下文记忆进行交互？ |
| **常见失分点** | ❌ 只支持单轮"一问一答"<br>❌ 没有使用会话历史功能 |
| **优化动作** | ✅ 使用chatflow的conversation_history_list节点<br>✅ 设计多轮交互场景（如：面试官连续提问→评估→给建议）<br>✅ 加入Question节点收集用户输入<br>✅ 多模态输入（图片+文字） |

### 指标9：提示词设计质量（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 有效地与大模型交互，体现人机协同的核心能力 |
| **检查要点** | 1. 提示词是否有清晰的角色设定？<br>2. 是否有明确的输出格式要求？<br>3. 是否包含了防止幻觉的约束？<br>4. 不同节点的提示词是否有差异化设计？ |
| **常见失分点** | ❌ 使用默认提示词（"你是一个AI助手"）<br>❌ 提示词过于简单，没有角色和行为约束<br>❌ 所有节点使用相同或相似提示词 |
| **优化动作** | ✅ 每个LLM节点写独立、详细的提示词<br>✅ 包含：角色设定 + 任务说明 + 输出格式 + 约束条件 + 示例<br>✅ 针对竞赛场景优化（如：面向中小学生的语言风格） |

## 四、扩展性（10%）

### 指标10：可扩展到特定设备（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 不仅用于电脑及手机端，可嵌入其它智能设备端，形成软硬融合 |
| **检查要点** | 1. 是否可连接外部设备？（传感器、摄像头、智能音箱等）<br>2. 是否在方案中说明了硬件扩展的可能性？<br>3. 小学组是可选项，高中组建议争取 |
| **常见失分点** | ❌ 完全没有考虑硬件扩展<br>❌ 方案中提到"可以用于XXX设备"但没有具体描述 |
| **优化动作** | ✅ 在方案设计文档中增加"硬件扩展设想"章节<br>✅ 描述2-3个具体的硬件扩展场景<br>✅ 即使不实际连接硬件，也说明扩展路径 |

### 指标11："输出校验"功能（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 包含对大模型输出的验证逻辑，防止幻觉导致系统崩溃 |
| **检查要点** | 1. 是否有代码节点校验LLM输出？<br>2. 校验了哪些内容？（格式、内容安全性、必填字段）<br>3. 校验失败后是否有兜底处理？ |
| **常见失分点** | ❌ 完全没有输出校验<br>❌ 有校验但校验失败后系统报错/无响应<br>❌ 只校验了表面格式，没校验内容合理性 |
| **优化动作** | ✅ 至少1个代码节点做输出校验<br>✅ 校验内容：JSON格式、字数限制、敏感词过滤、必填字段完整性<br>✅ 校验失败后：重试机制（最多N次）或兜底输出<br>✅ 参考案例5的output_validation子流程 |
| **阶段1预评估** | 这是容易做到的加分项！每个智能体都能加 |

## 五、传播性（15%）

### 指标12：汇报过程完整流畅（10%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 表达清楚、流畅，有感染力和说服力，条理性、逻辑性强 |
| **检查要点** | 1. 视频展示是否在5分钟内说清楚了？（做什么→怎么做→效果如何→反思）<br>2. 是否展示了真实操作过程？<br>3. 演讲者是否自信、流畅？ |
| **常见失分点** | ❌ 视频超时或内容空洞<br>❌ 全程看稿子念<br>❌ 没有展示智能体的实际操作<br>❌ 逻辑混乱，跳跃 |
| **优化动作** | ✅ 视频按"问题→方案→演示→效果→反思"五段结构<br>✅ 提前写脚本，练习2-3遍<br>✅ 必须展示智能体的实际运行（屏幕录制）<br>✅ 时间控制在4分半左右，留30秒缓冲 |

### 指标13：有使用记录（5%）

| 维度 | 内容 |
|------|------|
| **满分标准** | 开发的应用有被其他用户使用的记录或用户回访记录 |
| **检查要点** | 1. 是否有其他同学/老师使用过你的智能体？<br>2. 是否有使用截图或反馈截图？<br>3. 小学组有1-2条记录即可，初中/高中组建议3-5条 |
| **常见失分点** | ❌ 完全没有使用记录<br>❌ 使用记录看起来是自己测试的（同一台设备/同一个账号）<br>❌ 使用记录只有"好用！"没有具体反馈 |
| **优化动作** | ✅ 至少找2-3位同学试用<br>✅ 截图保存对话记录<br>✅ 收集具体反馈文字<br>✅ 在PPT和报告中展示使用记录和反馈 |

## 八、快速优化速查（阶段5使用）

| 如果这项偏低... | 最快优化方式 |
|----------------|-------------|
| 主题有新意 | 调整定位，找更细分的服务场景 |
| 多学科融合 | 增加1-2个学科元素，方案文档中标注 |
| 工作流开发 | 适当拆分节点、添加校验节点 |
| 多模态调用 | 增加图片输入或语音交互节点 |
| 多智能体协同 | 把一个LLM拆成2-3个不同角色的LLM |
| 智能人机交互 | 增加Question节点或conversation_history |
| 提示词设计 | 为每个LLM节点写详细的角色+格式+约束 |
| 输出校验 | 增加1个代码校验节点（最容易加的优化） |
| 设备扩展 | 方案文档中增加硬件扩展设想章节 |
| 使用记录 | 找同学试用并截图 |
