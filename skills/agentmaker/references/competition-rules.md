# 竞赛规则速查手册

> 基于"第九届全国青少年人工智能创新挑战赛·AI智能体开发专项赛"官方规则整理（2026年5月）  
> 本文档供 `/agentmaker` Skill 各阶段快速查询使用

---

## 一、参赛条件速览

| 项目 | 要求 |
|------|------|
| 参赛资格 | 在校小学、初中、高中、中专或职高学生 |
| 组别划分 | 小学低年级组（1-3年级）、小学高年级组（4-6年级）、初中组（7-9年级）、高中组（含中专、职高） |
| 参赛形式 | 个人赛或团队赛（2-3人/队），按最高学段划分组别 |
| 指导老师 | 不超过2人 |
| 报名时间 | 2026年5月6日—5月31日 |
| 选拔赛 | 2026年6月5日—7月5日 |

---

## 二、一票否决项（必须避免！）

以下3类作品**直接淘汰，不予评审**：

| 类型 | 判定标准 | 避坑建议 |
|------|----------|----------|
| **AI 直接生成** | 使用生成式AI直接生成的参赛思路、解决方案、文生文/图/视频/PPT等 | 作品必须体现学生自己的思考和设计过程，AI只是工具 |
| **纯代码编程** | 仅通过图形化、C++、Python等编写，未体现工作流和大模型决策特征 | 必须基于工作流开发，包含大模型节点的"智能决策" |
| **人文艺术类** | 小故事、小论文、音乐类、美术类、语言类、非遗类 | 主题必须与科技应用相关，有实际功能 |

### 阶段1收敛环节的"一票否决过滤器"检查清单：
- [ ] 作品是否体现学生独立思考和设计？（而非AI生成）
- [ ] 是否基于工作流开发？（而非纯代码）
- [ ] 是否属于科技应用类？（而非人文艺术）
- [ ] 是否体现"智能决策"？（而非简单规则匹配）

## 三、硬性技术要求

| 要求项 | 小学组 | 初中组/高中组 |
|--------|:------:|:------------:|
| **工作流节点数** | ≥ 5个 | ≥ 10个 |
| **判断节点** | 必须包含基于大模型输出的判断节点 | 必须包含基于大模型输出的判断节点 |
| **开发方式** | 必须基于工作流方式开发 | 必须基于工作流方式开发 |
| **开发平台** | 不限（Coze、Dify等均可） | 不限（Coze、Dify等均可） |
| **硬件结合** | 鼓励 | 鼓励（软硬结合加分） |

### 关键说明
- **"节点"** 指工作流中的处理节点（LLM节点、代码节点、判断节点、知识库节点等），开始/结束节点不计入
- **"判断节点"** 特指基于大模型输出进行条件路由的选择器/条件节点，必须体现AI的智能决策能力，单纯的 if-else 用户输入判断不满足要求
- 初学者建议目标：小学组 5-7 个节点，初中组 10-12 个节点，高中组 12-15 个节点

## 四、评分标准：5大维度13项指标

### 4.1 详细评分表

| 评分维度 | 序号 | 指标 | 权重 | 描述 |
|---------|:---:|------|:---:|------|
| **创新性 (25%)** | 1 | 主题有新意 | 15% | 项目主题具有新意，避免大众化，杜绝雷同，实用价值突出 |
| | 2 | 多学科融合 | 10% | 开发及设计中涉及多个学科的融合 |
| **完整性 (20%)** | 3 | 开发流程完整性 | 10% | 报告中包含需求分析、系统设计、开发与测试完整流程 |
| | 4 | 提交内容和资源完整性 | 10% | 提交内容包括报告文件及报告视频等完整资源 |
| **先进性 (30%)** | 5 | 工作流开发 | 10% | 项目应用工作流开发实现 |
| | 6 | 多模态调用 | 5% | 项目设计开发流程中包含一个以上大模型调用 |
| | 7 | 多智能体协同 | 5% | 系统中包含1个以上智能体，多智能体协同 |
| | 8 | 智能人机交互 | 5% | 系统包含一种以上智能人机交互 |
| | 9 | 提示词设计质量 | 5% | 有效地与大模型交互，体现人机协同的核心能力 |
| **扩展性 (10%)** | 10 | 可扩展到特定设备 | 5% | 不仅用于电脑及手机端，可嵌入其它智能设备端，形成软硬融合 |
| | 11 | "输出校验"功能 | 5% | 包含对大模型输出的验证逻辑（如正则表达式匹配、关键词过滤），防止幻觉导致系统崩溃 |
| **传播性 (15%)** | 12 | 汇报过程完整流畅 | 10% | 表达清楚、流畅，有感染力和说服力，条理性、逻辑性强 |
| | 13 | 有使用记录 | 5% | 开发的应用有被其他用户使用的记录或用户回访记录 |

### 4.2 维度权重汇总

| 维度 | 总权重 | 重要性排序 |
|:---|:---:|:---:|
| **先进性** | 30% | ★★★★★ 最高 |
| **创新性** | 25% | ★★★★ |
| **完整性** | 20% | ★★★ |
| **传播性** | 15% | ★★ |
| **扩展性** | 10% | ★ |

### 4.3 得分策略建议

| 策略 | 说明 |
|------|------|
| **保底得分**（必须） | 工作流开发(10%) + 完整性(20%) = 30%，这是基础 |
| **进阶得分**（重点） | 主题有新意(15%) + 多模态(5%) + 多智能体(5%) + 智能交互(5%) = 30% |
| **锦上添花** | 多学科融合(10%) + 扩展设备(5%) + 输出校验(5%) + 使用记录(5%) = 25% |
| **汇报展示** | 汇报流畅(10%) + 提示词质量(5%) = 15% |

## 五、交付件清单

| 序号 | 交付件 | 内容要求 |
|:---:|--------|----------|
| 1 | 源代码或网络链接 | AI智能体系统的源代码或网络链接（Coze Bot链接） |
| 2 | 附件1 项目基本信息表 | 填写完整（含成员姓名、学校、指导老师等） |
| 3 | 项目报告 PPT | 需求分析、系统设计描述（含智能体系统设计框图、工作流设计截图）、开发与测试过程、开发反思 |
| 4 | 报告视频（MP4） | 个人身份认证、作品路演、过程性记录（5分钟展示 + 2分钟评委提问） |
| 5 | 过程文档/照片/视频 | 开发过程记录材料 |

### 文件命名规范
> **组别-省-市-学校名称-选手名称-作品名称**

不按规定命名视为无效作品！

### 视频规格
- 格式：MP4
- 分辨率：≥ 1280×720
- 大小：< 200MB

## 六、决赛流程

- **形式**：线上举办
- **内容**：选拔赛作品基础上再创作
- **展示时间**：5分钟展示 + 2分钟评委提问 = 7分钟总计
- **展示形式**：不限（语言阐述、PPT演示、播放视频、场景展示等）

|------|:----------:|:----------:|:-----:|:-----:|

| 软硬结合 | 可选 | 可选 | 加分项 | 加分项 |
| 输出校验 | 可选 | 可选 | 建议 | 建议 |
| 推荐节点数目标 | 5-7 | 5-8 | 10-12 | 12-15 |

## 八、常见违规与避坑

| 常见问题 | 后果 | 解决方案 |
|----------|------|----------|
| 节点数不达标 | 初审不通过 | 适当拆分复杂节点、添加校验节点 |
| 无大模型判断节点 | 扣分严重（先进性维度不达标） | 至少1个条件选择器基于LLM输出做路由 |
| 工作流截图缺失 | 完整性扣分 | 阶段6交付前逐一检查截图 |
| AI代写报告 | 一票否决 | 报告必须体现学生视角和过程记录 |
| 命名不规范 | 视为无效作品 | 严格按照 `组别-省-市-学校-选手-作品` 格式 |
| 视频超时/超限 | 扣分 | 严格控制在5分钟内，<200MB |
| 主题雷同 | 创新性维度大量失分 | 阶段1充分发散，避免常见主题 |
