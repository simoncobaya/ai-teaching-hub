# Coze（扣子）平台搭建通用SOP

> **本文档基于Coze平台 2026-05 版本编写**  
> 如Coze平台有重大更新，以平台实际界面为准  
> 本文档为通用搭建流程指南，不绑定特定案例

---

## 一、平台概述

Coze（扣子）是字节跳动推出的AI智能体开发平台，支持通过可视化工作流（Workflow）搭建AI智能体。

### 当前版本可用功能（2026年5月）

| 功能模块 | 说明 |
|----------|------|
| **智能体（Bot）** | 基础对话单元，支持文本/语音/图片交互 |
| **Chatflow** | 面向对话场景的工作流类型，支持会话历史和流式输出 |
| **Workflow** | 通用工作流，可作为Subflow被Chatflow或其他Workflow调用 |
| **知识库（Knowledge）** | RAG检索增强，支持文档上传和语义搜索 |
| **插件（Plugin）** | 调用外部API扩展能力 |
| **变量（Variable）** | 全局变量存储与传递 |

### 常用节点类型

| 节点类型 | 用途 | 使用场景 |
|----------|------|----------|
| **大模型（LLM）** | AI文本生成/分析/判断 | 意图识别、内容生成、答案评判等核心智能处理 |
| **选择器（Condition）** | 条件分支路由 | 根据变量值选择不同处理路径 |
| **代码（Code）** | JavaScript/Python代码执行 | 格式校验、数据转换、规则判断 |
| **知识库（Knowledge）** | 检索知识库文档 | 需要专业知识库支持的场景 |
| **子流程（Subflow）** | 调用另一个Workflow | 复用逻辑、拆分复杂流程 |
| **循环（Loop）** | 重复执行 | 需要重试或迭代的场景 |
| **变量聚合（Variable Merge）** | 合并多个分支输出 | 多分支汇流到同一出口 |
| **问答（Question）** | 向用户提问并等待回复 | 需要收集用户输入 |
| **文本拼接（Text）** | 文本格式化拼接 | 固定格式的文本拼接 |
| **设置变量（Set Variable）** | 设置变量的值 | 循环中更新变量 |
| **终止循环（Break）** | 终止循环 | 条件满足时跳出循环 |
| **查询会话历史** | 获取对话历史 | Chatflow中获取上下文 |
| **图像识别** | 多模态图片理解 | 图片输入场景 |

### 可用模型（2026年5月）

| 模型 | 特点 | 推荐场景 |
|------|------|----------|
| 豆包·1.8·深度思考 | 推理能力强，JSON输出稳定 | 意图识别、分析判断、知识讲解 |
| 豆包·2.0·lite | 速度快，轻量级 | 简单问答、固定格式输出 |
| DeepSeek-R1 | 开源推理模型 | 复杂推理任务 |
| 豆包·视觉 | 多模态理解 | 图片分析、OCR |

## 二、搭建流程总览

```
第1步：创建智能体（Bot）
第2步：配置人设与开场白
第3步：创建知识库（如需要）
第4步：搭建Subflow（从底层到顶层）
第5步：搭建主Workflow
第6步：搭建Chatflow入口
第7步：测试工作流
第8步：发布智能体
```

---

## 三、第1步：创建智能体（Bot）

### 3.1 创建Bot

1. 登录 Coze 平台（https://www.coze.cn/space）
2. 点击「创建」→ 选择「创建智能体」
3. 填写基本信息：
   - **名称**：智能体的名字（建议格式：`功能描述-智能体名称`）
   - **简介**：一句话描述智能体做什么
   - **头像**：上传相关图标

> [此处插入截图]创建智能体界面

### 3.2 配置人设

在「人设与回复逻辑」中编写系统提示词（System Prompt），这是智能体的"人格设定"：

```markdown
你是「XXX」系统的AI助手。你的职责是：
1. [功能1描述]
2. [功能2描述]
3. [功能3描述]

你的语气要[风格描述]。
当[场景A]时，[行为描述]；当[场景B]时，[行为描述]。
```

> [此处插入截图]人设配置界面

### 3.3 配置开场白

在「开场白」中设置用户首次打开时的欢迎语：

```markdown
欢迎来到「XXX」！

我是你的[功能定位]，可以帮你：
📌 [功能1]
📌 [功能2]
📌 [功能3]

请告诉我：[引导用户输入的内容]
```

## 四、第2步：创建知识库（如需要）

如果你的智能体需要专业知识支持（如学科知识、行业规范等），需要创建知识库。

### 4.1 创建知识库

1. 在左侧导航选择「知识库」→ 点击「创建知识库」
2. 设置知识库名称和描述
3. 上传知识文档（支持 .md, .txt, .pdf, .docx 等格式）
4. 选择分段策略（推荐：自动分段，分段长度 800-1000 字符）
5. 等待处理完成（通常1-3分钟）

### 4.2 多知识库场景

如果涉及多个领域，建议创建多个独立知识库：
- 按学科分类（数学知识库、语文知识库...）
- 按功能分类（FAQ知识库、专业文档知识库...）
- 按层级分类（基础知识库、进阶知识库...）

> [此处插入截图]知识库创建界面

### 4.3 知识库在LLM节点中使用

在LLM节点中，点击「关联知识库」即可挂载。注意：
- 一个LLM节点可以关联多个知识库
- 可以通过选择器按条件路由到不同知识库节点
- 知识库节点的"搜索策略"推荐使用「混合」（结合语义和关键词检索）

## 五、第3步：搭建工作流（通用方法）

### 5.1 搭建顺序原则

> **从底层到顶层搭建**：先搭最底层的Subflow（叶子节点），再逐层向上组合。

**为什么这样搭？**
- 底层Subflow没有依赖，可以先独立测试
- 上层调用下层时，下层已经验证通过，排查问题更容易
- 符合"先拆后合"原则

### 5.2 创建工作流

1. 在智能体编辑页 → 点击「工作流」标签
2. 点击「新建工作流」
3. 选择类型：
   - **Workflow**：通用工作流（可作为Subflow被调用）
   - **Chatflow**：对话型工作流（用户交互入口）
4. 设置名称和描述

### 5.3 配置开始节点

开始节点定义工作流的输入接口：

| 变量名 | 类型 | 是否必需 | 说明 |
|--------|------|:---:|------|
| [变量1] | String/Number/Boolean/Object/List | 是/否 | [说明] |

**类型选择建议**：
- 纯文本 → `String`
- 列表/数组 → `List`
- 复杂结构 → `Object`
- 布尔判断 → `Boolean`

### 5.4 添加LLM节点

LLM节点是工作流的核心。标准配置步骤：

1. 拖入「大模型」节点
2. 选择模型（推荐：豆包·1.8·深度思考）
3. 设置 responseFormat：JSON（需要结构化输出时）
4. 编写系统提示词（System Prompt）
5. 编写用户提示词（User Prompt，引用输入变量）
6. 如需知识库，点击「关联知识库」

**提示词编写模板**：

```markdown
[角色设定] 你是[角色名称]，负责[职责描述]。

[任务说明] 请根据以下输入，完成[具体任务]。

[规则约束]
1. [规则1]
2. [规则2]

[输出格式] 请严格输出JSON格式：
{
  "field1": "值",
  "field2": "值"
}
```

**用户提示词**（引用变量）：
```
输入内容：{{变量名1}}
上下文：{{变量名2}}
```

### 5.5 添加选择器（Condition）节点

选择器基于LLM输出实现条件分支。配置步骤：

1. 拖入「选择器」节点
2. 配置条件分支：

| 条件 | 表达式 | 目标 |
|------|--------|------|
| 条件1 | `{{节点名.变量}}` == "值" | 分支A |
| 条件2 | `{{节点名.变量}}` 包含 "关键词" | 分支B |
| ELSE | 其他情况 | 默认分支 |

**条件表达式类型**：
- 等于：`==`、`!=`
- 包含：使用 `包含` 操作符
- 数值比较：`>`、`<`、`>=`
- OR逻辑：多个条件使用OR

> ⚠️ **竞赛要求**：必须包含至少1个基于LLM输出的选择器（不是基于用户输入的简单if-else）

### 5.6 添加代码节点

代码节点用于确定性逻辑处理。支持 JavaScript 和 Python。

**代码节点模板（JavaScript）**：

```javascript
async function main({ params }: Args): Promise<Output> {
    const input = params.input;
    
    // 1. 数据校验
    if (!input || input.length < 1) {
        return { valid: false, error: "输入为空" };
    }
    
    // 2. 数据处理
    const result = /* 处理逻辑 */;
    
    // 3. 返回结果
    return { valid: true, output: result };
}
```

**代码节点常用场景**：
- LLM输出格式校验（检查JSON完整性、必填字段）
- 内容安全过滤（敏感词检测）
- 数据格式转换（拼接、解析）
- 简单规则判断（不需要LLM的确定性判断）

### 5.7 添加子流程（Subflow）节点

1. 拖入「子流程」节点
2. 选择目标Workflow
3. 映射传入参数（将当前工作流的变量传递给子流程的输入变量）
4. 子流程执行完毕后，其输出变量可在后续节点引用

**参数映射示例**：
```
当前工作流变量 "user_input" → 子流程输入 "input_text"
当前工作流变量 "level" → 子流程输入 "difficulty"
```

### 5.8 添加循环（Loop）节点

1. 拖入「循环」节点
2. 设置最大循环次数（推荐：10次，防止死循环）
3. 设置循环类型：infinite（手动终止）
4. 配置循环变量（初始值）

**循环内的终止条件**：
- 校验通过 → 终止循环
- 超过重试次数 → 终止循环（兜底）
- 其他条件 → 继续循环

**兜底逻辑**：如果循环达到最大次数仍未满足条件，需要有兜底输出（如固定文案），确保系统不会"卡住"。

### 5.9 添加变量聚合节点

当多个分支汇流到同一出口时使用：

1. 拖入「变量聚合」节点
2. 将各分支的输出都连到该节点
3. 配置聚合变量名
4. 结束节点引用聚合变量

### 5.10 配置结束节点

结束节点定义工作流的输出：

| 变量名 | 类型 | 引用来源 |
|--------|------|----------|
| [输出变量1] | String/Object | [引用某个节点的输出] |
| [输出变量2] | String | [引用某个节点的输出] |

**Chatflow的结束节点**：需要开启「流式输出」（streamingOutput），以提升用户体验。

## 六、第4步：Chatflow入口搭建要点

Chatflow是用户交互的入口，有特殊配置：

### 6.1 特殊节点

Chatflow自动提供：
- **开始节点**：`USER_INPUT`（用户输入）、`CONVERSATION_NAME`（会话名称）
- **查询会话历史**：获取最近N轮对话（rounds参数，建议30轮）

### 6.2 标准Chatflow结构

```
[开始] → [查询会话历史] → [主Workflow Subflow] → [结束(流式输出)]
```

参数映射：
- `USER_INPUT` → Subflow的 `user_input`
- `查询会话历史.messageList` → Subflow的 `session_history`

### 6.3 绑定Chatflow到Bot

> ⚠️ **关键步骤**：Chatflow搭建完毕后，必须绑定到Bot才会生效，否则用户对话时不会触发工作流。

1. 回到Bot编辑页面
2. 在「工作流」标签中，点击「添加工作流」
3. 选择刚刚搭建的 Chatflow（如 `trade_security_chat`）
4. 保存Bot配置

至此，用户与Bot的对话才会进入 Chatflow 的处理流程。

## 七、第5步：测试与调试

### 7.1 单节点测试

1. 选中单个LLM节点 → 点击「试运行」
2. 输入测试数据
3. 检查输出格式和内容是否符合预期

### 7.2 Subflow独立测试

每个Subflow搭建完毕后，独立运行测试：
- 输入各种正常和边界值
- 确认输出格式正确
- 确认错误场景有兜底处理

### 7.3 全链路测试

在Chatflow中完整测试，覆盖所有分支：

| 测试类型 | 测试内容 | 验证点 |
|----------|----------|--------|
| 主路径测试 | 最常见的用户输入 | 完整流程走通 |
| 分支测试 | 每个分支的触发条件 | 各分支都能正确到达 |
| 边界测试 | 空输入、乱码、超长文本 | 有兜底处理，不报错 |
| 错误恢复 | LLM输出异常 | 有重试或兜底处理 |

### 7.4 常见调试技巧

| 问题 | 解决方法 |
|------|----------|
| LLM输出不是JSON | 在系统提示词中强调"严格输出JSON"，设置responseFormat为JSON |
| 选择器条件不生效 | 先用LLM节点输出原始结果确认格式，再设条件表达式 |
| 子流程调用失败 | 检查输入参数名称和类型是否匹配子流程开始节点的定义 |
| 知识库检索不到 | 检查文档是否上传成功，关键词是否匹配 |
| 循环死循环 | 检查终止条件，确保有兜底（如 index>N 的强制终止） |
| Chatflow无法触发 | 确认Chatflow已绑定到智能体的「工作流」设置中 |

## 八、第6步：发布

1. 完成全部测试后，点击右上角「发布」
2. 选择发布渠道：「网页链接」最常用
3. 生成分享链接
4. 建议先小范围试用，收集反馈后再正式提交

### 发布前检查清单

- [ ] 所有工作流名称清晰有意义
- [ ] 开场白已配置
- [ ] 人设与回复逻辑已编写
- [ ] **Chatflow已绑定到Bot的工作流配置**
- [ ] 全链路测试通过
- [ ] 各分支路径验证通过
- [ ] 错误场景有兜底处理
- [ ] 知识库已关联（如需要）

## 九、变量命名规范建议

| 变量名 | 类型 | 命名建议 |
|--------|------|----------|
| user_input / cleaned_input | String | 用户输入（原始/清洗后） |
| session_history | List(Object) | 会话历史 |
| intent / input_type | String | 意图分类结果 |
| result / output | String | 处理结果 |
| is_valid / success | Boolean | 校验/操作是否成功 |
| error_message | String | 错误信息 |
| feedback | String | 用户反馈内容 |

## 十、Coze vs CodeBuddy Skill 的定位

| 维度 | Coze Bot | CodeBuddy Skill |
|------|----------|-----------------|
| 运行环境 | Coze云端平台 | CodeBuddy本地IDE |
| 搭建方式 | 可视化拖拽 | 编写SKILL.md文本 |
| 测试速度 | 修改→发布→测试，1-2分钟 | 保存即生效，秒级测试 |
| 最终提交 | ✅ 竞赛需要提交Coze Bot链接 | ❌ 仅用于开发和验证 |
| 适用阶段 | 阶段4-6（正式搭建+交付） | 阶段3（快速原型验证） |

**推荐策略**：阶段3先用CodeBuddy Skill快速验证工作流逻辑，确认无误后阶段4再到Coze"照图纸施工"。
